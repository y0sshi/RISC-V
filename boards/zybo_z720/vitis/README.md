# Zybo Z7-20 real-HW bring-up (Vitis / boot side)

Offline (no-board) preparation for booting the RISC-V SoC bitstream on real Zybo
Z7-20 hardware. The Vivado bitstream side is in `../vivado/`; this directory holds
the PS-side boot artifacts (XSA -> FSBL -> BOOT.bin) and the JTAG bring-up script.

All Xilinx tools must run from **PowerShell** with **absolute** paths (Bash/MSYS
path translation crashes Vivado synth and the Vitis server).

## Clock / UART facts baked into the design (read first)

- PL clock `FCLK_CLK0` = **25 MHz** (`build_zybo.tcl` `PL_FREQMHZ=25`).
- `rv_soc` `MTIME_DIV=1` => CLINT `mtime` ticks at **25 MHz** on hardware.
- The 16550 bit period is `divisor*16` PL clocks; the 8250 driver programs
  `divisor = clock-frequency/(16*baud)`, so `actual_baud = PL/(divisor*16) =
  25e6*baud/clock-frequency`.

Therefore the **real-HW device tree** (`docs/opensbi/rv_soc_hw.dts`,
`tests/linux/rv_soc_linux_hw.dts`) differs from the sim DTS in three constants:

| DT property | sim (`*_lo.dts`) | real HW (`*_hw.dts`) | why |
|---|---|---|---|
| `timebase-frequency` | 1000000 | **25000000** | mtime really runs at 25 MHz (sim slows it via `BOOT_MTIME_DIV=64`); wrong value = 25x time error + HZ-tick livelock risk |
| serial `clock-frequency` | 1843200 | **25000000** | must equal the PL clock or the baud divisor is wrong |
| serial `current-speed` | 115200 | **57600** | 25 MHz can't make a clean 115200 (+/-4%); 57600 -> divisor 27 -> 57870 baud = +0.47% |

Use a **57600 8N1** terminal on hardware.

## prep-A/B/C (done offline)

- **prep-A** real-HW DTS: `docs/opensbi/rv_soc_hw.dts`, `tests/linux/rv_soc_linux_hw.dts`.
- **prep-B** export XSA from the built impl (no re-impl):
  ```
  & "E:\Tools\Xilinx\Vivado\2024.2\bin\vivado.bat" -mode batch `
      -source E:\work\git\RISC-V.git\boards\zybo_z720\vivado\export_xsa.tcl `
      *> ...\export_xsa.log 2>&1
  ```
  -> `../vivado/rv_riscv_zybo/rv_riscv_zybo.xsa` (ps7_init + bitstream). The full
  `build_zybo.tcl bit` flow now also emits the XSA automatically.
- **prep-C** FSBL + BOOT.bin:
  ```
  & "E:\Tools\Xilinx\Vitis\2024.2\bin\vitis.bat" -s `
      E:\work\git\RISC-V.git\boards\zybo_z720\vitis\fsbl.py *> ...\fsbl.log 2>&1
  .\make_bootbin.ps1 -Firmware <fw_payload_hw.bin>   # default = sim fw_payload_lo.bin
  ```
  - FSBL is auto-generated by the standalone platform build ->
    `ws/zybo_plat/export/zybo_plat/sw/boot/fsbl.elf` (32-bit ARM, ps7_init).
  - `make_bootbin.ps1` writes `boot.gen.bif` (firmware **before** bitstream, raw
    `.bin` with `[load=0x00200000]` and no exec addr) and runs bootgen. Verified
    layout: FSBL(exec OCM) -> firmware@0x200000 (data, no A9 handoff) -> bitstream
    last, so the RISC-V core leaves reset with firmware already resident in DDR.

## Remaining (needs the board)

- **prep-E** rebuild the firmware with the real-HW DTS, then regenerate BOOT.bin:
  ```
  docker run --rm -v "E:/work/git/RISC-V.git:/workspace" -w /workspace/tests/opensbi \
    riscof_run:latest bash -c \
    "FW_BASE=0x00200000 DTS=/workspace/docs/opensbi/rv_soc_hw.dts OUT=fw_payload_hw bash build.sh"
  .\make_bootbin.ps1 -Firmware E:\work\git\RISC-V.git\tests\opensbi\work\fw_payload_hw.bin
  ```
  (Linux: `tests/linux/build.sh` with `DTS=.../rv_soc_linux_hw.dts OUT=fw_payload_linux_hw`,
  CRLF-strip per CLAUDE.md.)
- **prep-D / P1** JTAG bring-up (script to author when at the board; uses the
  debug side of xsct, not the project flow): `connect` -> select APU ->
  `source ps7_init.tcl; ps7_init; ps7_post_config` (PS/DDR up) -> `fpga -file
  bd_riscv_wrapper.bit` -> `dow fw_payload_hw.elf` (loads to 0x200000) -> release
  the PL reset so the core boots from 0x200000. `ps7_init.tcl` is inside the XSA.
- **P1-c** observe the OpenSBI banner on the **Pmod JC** USB-UART (FPGA TX=V15 ->
  adapter RX, FPGA RX=W15 <- adapter TX, common GND), **57600 8N1**.
